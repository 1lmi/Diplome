# SC restaurant

Единая документация проекта SC restaurant. Этот файл описывает текущее состояние системы и служит основным источником правды по функциональности, архитектуре и технологиям.

Проект состоит из четырёх основных частей:

1. Backend API
2. Публичный сайт
3. Админ-сайт
4. Мобильное приложение

Все клиентские части работают с одним backend и одной моделью данных: меню, категории, заказы, статусы, настройки витрины, пользователи, адреса и изображения.

## Общая архитектура

- Backend: `FastAPI` + `SQLite`
- Публичный сайт и админ-сайт: `React 19` + `TypeScript` + `Vite`
- Мобильное приложение: `Expo` + `React Native`
- Авторизация: bearer token + таблица серверных сессий
- Изображения: файловое хранилище `uploads/`, выдача через `/static/...`
- Карты:
  - web: Яндекс Карты используются в адресном flow
  - mobile: экран адреса использует `WebView + hosted Yandex Maps wrapper page`

## 1. Backend

### Назначение

Backend хранит и обслуживает всё состояние системы: каталог, настройки витрины, пользователей, адреса, заказы, статусы заказов, изображения и push-токены.

### Что умеет backend

#### Публичные функции

- отдаёт настройки витрины через `GET /settings`
- отдаёт список статусов заказа через `GET /order-statuses`
- отдаёт категории через `GET /categories`
- отдаёт меню и карточки товаров через `GET /menu` и `GET /menu/{item_id}`
- принимает новые заказы через `POST /orders`
- позволяет отследить заказ по `order_id + phone` через `GET /orders/track`
- позволяет получить детали заказа через `GET /orders/{order_id}`

#### Пользовательские функции

- регистрация через `POST /auth/register`
- вход через `POST /auth/login`
- получение текущего пользователя через `GET /auth/me`
- обновление профиля через `PUT /me`
- выход через `POST /auth/logout`
- получение списка адресов через `GET /me/addresses`
- создание, редактирование и удаление адресов через `POST /me/addresses`, `PATCH /me/addresses/{address_id}`, `DELETE /me/addresses/{address_id}`
- получение истории заказов через `GET /me/orders`
- регистрация и удаление Expo push token через `PUT /me/push-token` и `DELETE /me/push-token`

#### Административные функции

- выдаёт агрегированное меню для админки через `GET /admin/menu`
- создаёт, редактирует и удаляет категории
- создаёт, редактирует и удаляет товары и их варианты
- принимает загрузку изображений через `POST /admin/upload-image`
- отдаёт список заказов для операционной панели через `GET /admin/orders`
- меняет статус заказа через `PATCH /orders/{order_id}/status`
- обновляет витринные настройки через `PUT /admin/settings`

### Как это работает

- Все запросы идут в один FastAPI сервис из [main.py](/c:/Users/mg853/VSCode_prodj/Diplome/main.py).
- База данных `SQLite` хранится в `sc-restaurant.db`.
- При старте backend применяет миграции и добивает недостающие поля, таблицы и базовые значения.
- Статусы заказов сидируются автоматически.
- Если администратора нет, backend создаёт дефолтного admin-пользователя.
- Изображения складываются в `uploads/` и отдаются по `/static/...`.

### Авторизация и роли

- Пароли хешируются через `PBKDF2-HMAC-SHA256`
- Сессии хранятся на сервере в таблице `sessions`
- Токен клиента это bearer token, связанный с записью сессии
- Роль администратора определяется флагом `is_admin`
- Админские endpoint-ы доступны только пользователям с этой ролью

### Какие данные использует backend

Ключевые сущности:

- `users`
- `sessions`
- `categories`
- `products`
- `product_sizes`
- `orders`
- `order_items`
- `order_statuses`
- `order_status_history`
- `settings`
- `user_addresses`
- `push_tokens`

### Особенности текущей реализации

- База данных локальная, на `SQLite`
- CORS сейчас открыт достаточно широко для удобства разработки
- Статусы заказов и transitions контролируются на backend
- Push-уведомления отправляются через Expo push API

## 2. Сайт

### Назначение

Публичный сайт это клиентская витрина для заказа еды: просмотр меню, выбор товаров, корзина, checkout, профиль и история заказов.

### Технологии

- `React 19`
- `TypeScript`
- `Vite`
- `react-router-dom`
- контексты авторизации и корзины
- локальные UI-компоненты в `src/components`

### Основные функции

#### Каталог и меню

- главная страница показывает меню по категориям
- есть шапка с навигацией, категориями и кнопкой корзины
- карточка товара открывается в модальном окне
- поддерживаются товары с размерами и вариантами
- товары можно добавлять в корзину прямо из каталога

#### Корзина и оформление

- есть отдельная страница корзины `/cart`
- checkout расположен на `/checkout`
- поддерживаются `delivery` и `pickup`
- после оформления пользователь попадает на экран успешного заказа

#### Аккаунт и заказы

- есть регистрация и логин
- профиль расположен на `/profile`
- в профиле доступны личные данные, история заказов и адреса
- есть экран деталей отдельного заказа `/orders/:orderId`

#### Маршруты сайта

Основные web routes:

- `/`
- `/cart`
- `/checkout`
- `/checkout/success/:orderId`
- `/orders/:orderId`
- `/profile`

### Как работает пользовательский flow

- пользователь открывает каталог
- выбирает товар и добавляет его в корзину
- переходит в checkout и оформляет заказ
- после оформления может перейти к деталям заказа
- после авторизации получает доступ к профилю, адресам и истории заказов

### Какие данные использует сайт

- настройки витрины
- категории и меню
- текущая корзина
- текущий пользователь
- список адресов пользователя
- история заказов и детали заказа

### Особенности текущей реализации

- сайт и админка живут в одном React-приложении
- вход в админку зависит от `is_admin` у пользователя
- в публичной части используются витринные настройки из backend
- Яндекс-карта участвует в адресных сценариях на стороне web

## 3. Админ-сайт

### Назначение

Админ-сайт это операционная и контентная панель для управления витриной, каталогом и заказами.

Он доступен внутри web-приложения по маршруту `/admin/*`.

### Технологии

- тот же `React + TypeScript + Vite`, что и у публичного сайта
- `react-router-dom` для вложенных admin routes
- `@dnd-kit/*` для reorder категорий и позиций
- `recharts` для статистики

### Основные разделы

#### Dashboard

- сводка по системе и заказам
- редактирование витринных настроек
- управление hero-текстами и контактной информацией

#### Статистика

- страница `/admin/stats`
- статистика строится на frontend по загруженным заказам и статусам
- показывает базовые графики и срезы без отдельного server-side analytics endpoint

#### Управление категориями

- создание категорий
- редактирование названия, описания, порядка и видимости
- удаление категорий
- ручной reorder

#### Управление меню

- создание товаров
- редактирование товара через modal-форму
- работа с размерами и вариантами
- скрытие и удаление товаров
- reorder позиций внутри категорий
- загрузка изображений товаров

#### Заказы

- текущие заказы
- история заказов
- смена статуса заказа по допустимым переходам
- обновление данных вручную кнопкой refresh

### Маршруты админки

Основные admin routes:

- `/admin/dashboard`
- `/admin/stats`
- `/admin/categories`
- `/admin/menu`
- `/admin/orders/current`
- `/admin/orders/history`

### Как работает админский flow

- администратор логинится обычным пользовательским аккаунтом
- frontend проверяет `user.is_admin`
- после этого становятся доступны admin routes
- админ управляет витриной, меню и заказами через один SPA-интерфейс

### Какие данные использует админка

- settings витрины
- menu и categories
- список заказов и статусов
- изображения товаров

### Особенности текущей реализации

- админка не является отдельным проектом, она встроена в основной сайт
- статистика пока не вынесена в отдельный backend analytics API
- refresh данных выполняется из client-side панели

## 4. Приложение

### Назначение

Мобильное приложение дублирует основной клиентский пользовательский сценарий: каталог, товар, корзина, checkout, профиль, адреса, история заказов и просмотр заказа.

### Технологии

- `Expo`
- `React Native`
- `expo-router`
- `@tanstack/react-query`
- `zustand`
- `expo-location`
- `expo-notifications`
- `react-native-webview`
- `expo-secure-store`
- `AsyncStorage`

### Основные экраны

По коду в `sc-restaurant-mobile/src/screens` приложение включает:

- `HomeScreen`
- `ProductScreen`
- `CartScreen`
- `CheckoutScreen`
- `SignInScreen`
- `SignUpScreen`
- `ProfileScreen`
- `ProfileEditScreen`
- `OrdersScreen`
- `ProfileOrdersScreen`
- `OrderScreen`
- `AddressFormScreen`

### Основные функции

#### Каталог

- экран каталога показывает категории и карточки товаров
- есть переключение способа получения заказа
- есть выбор и быстрый показ адреса доставки
- карточки товаров открывают экран товара
- внизу показывается cart bar

#### Товар

- отдельный экран товара
- выбор размера или варианта
- показ цены, состава и КБЖУ
- добавление в корзину

#### Корзина и checkout

- экран корзины показывает текущие позиции
- checkout собирает delivery method, адрес, время, оплату и комментарий
- заказ отправляется в backend и после этого очищает локальную корзину

#### Аккаунт

- вход и регистрация
- профиль с краткой карточкой пользователя
- редактирование профиля
- просмотр адресов
- просмотр последних заказов и переход к полному списку

#### Заказы

- список заказов
- экран одного заказа
- трекинг и повторный просмотр деталей

#### Адреса

- создание нового адреса
- редактирование существующего
- выбор адреса по карте
- выбор label `Дом / Работа / Другое`

### Экран адреса в mobile

Текущая реализация адресного экрана:

- карта открывается не через `react-native-maps`, а через `WebView`
- внутри `WebView` используется hosted страница `https://sc-delivery.ru/mobile-yandex-map.html`
- карта и поиск адреса завязаны на Яндекс Карты
- pin визуально остаётся React Native overlay поверх `WebView`
- координаты, zoom и результаты поиска синхронизируются через message bridge между RN и страницей карты

### Состояние и хранение данных

- корзина и checkout draft хранятся в `zustand`
- данные пользователя и auth-состояние хранятся в мобильном store
- запросы к backend идут через `react-query`
- tracking и часть пользовательских данных сохраняются локально
- push token регистрируется на backend отдельно

### Особенности текущей реализации

- mobile использует тот же backend API, что и сайт
- адресный flow в приложении уже переведён на Яндекс-карту
- сборка Android идёт через Expo/EAS
- часть старых экранов всё ещё содержит warning-ы линтера, но не ломает основную логику

## Общие данные и интеграции

### Общие интерфейсы

Все клиентские части используют общие сущности:

- категории
- товары и размеры
- заказы и статусы заказов

- заказы, адреса, статусы и каталог общие для всех клиентов
